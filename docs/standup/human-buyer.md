# Human Buyer Walkthrough

This runbook packages the live isolated Ethereum Sepolia environment into a
manual, operator-friendly path. The goal is to let a human install the buyer
CLI into a local sandbox, purchase compute from the live seller, verify SSH
access, and clean up afterward without ad hoc debugging.

This path assumes:

- the isolated live environment is already up
- `gcloud` is authenticated
- local secrets are present in `~/.config/web3-ops` and `~/.config/simple-market-service`
- the live buyer and seller agent IDs already exist in `prod-canary.env`

## Required Inputs

- `buyer.env`, `seller.env`, and `context.json` generated by `scripts/setup_human_buyer_sandbox.py`
- a reachable `AGENT_AUTH_URL` for both buyer and seller
- `SSH_PRIVATE_KEY_PATH` present in `~/.config/simple-market-service/prod-canary.env`
- a healthy registry, provisioning service, buyer agent, and seller agent

## 1. Open The Local Tunnel

In one terminal, inspect the exact tunnel command:

```bash
python scripts/start_human_buyer_tunnel.py command
```

In a dedicated terminal, open it:

```bash
python scripts/start_human_buyer_tunnel.py open
```

Back in your operator terminal, verify the forwarded endpoints:

```bash
python scripts/start_human_buyer_tunnel.py check
```

The human flow expects these local forwards:

- registry: `http://127.0.0.1:28080`
- provisioning: `http://127.0.0.1:28081`
- buyer agent: `http://127.0.0.1:28001`
- seller agent: `http://127.0.0.1:28002`

## 2. Build The Local Sandbox

Create the sandboxed buyer install and render the local env bundle:

```bash
python scripts/setup_human_buyer_sandbox.py \
  --shared-secrets-dir ~/.config/web3-ops \
  --local-secrets-dir ~/.config/simple-market-service \
  --sandbox-dir /tmp/market-human-buyer
```

This creates:

- `/tmp/market-human-buyer/venv/bin/market`
- `/tmp/market-human-buyer/buyer.env`
- `/tmp/market-human-buyer/seller.env`
- `/tmp/market-human-buyer/context.json`

The rendered `buyer.env` and `seller.env` carry the tunnel-facing `AGENT_URL`
values and the canonical `AGENT_AUTH_URL` values that the live agents verify.

## 3. Preflight Funding

Before creating orders, make sure the buyer has enough native ETH and WETH for
the live purchase path:

```bash
python scripts/pre_canary_fund.py \
  --shared-secrets-dir ~/.config/web3-ops \
  --local-secrets-dir ~/.config/simple-market-service
```

If the plan shows a top-up, apply it before continuing:

```bash
python scripts/pre_canary_fund.py \
  --shared-secrets-dir ~/.config/web3-ops \
  --local-secrets-dir ~/.config/simple-market-service \
  --apply
```

## 4. Seed A Seller Offer From Live Inventory

Use the helper to create a seller offer from the live advertised portfolio so
the order matches what the seller can actually fulfill:

```bash
python scripts/seed_human_seller_offer.py \
  --context-path /tmp/market-human-buyer/context.json \
  --amount 0.0001
```

Record the returned seller order ID.

Optional: if you want a specific live resource, pass `--resource-id`. If you
want to filter the portfolio without naming the exact resource, pass
`--gpu-model` and/or `--region`.

## 5. Purchase Compute As The Buyer

Match the seller offer from the buyer side:

```bash
/tmp/market-human-buyer/venv/bin/market order match <seller-order-id> \
  --registry-url http://127.0.0.1:28080 \
  --agent-url http://127.0.0.1:28001 \
  --auth-agent-url http://10.243.0.117:8000 \
  --env /tmp/market-human-buyer/buyer.env
```

Record the returned buyer order ID.

## 6. Wait For Provisioning And Verify SSH

Once you have the seller and buyer order IDs, wait for the create job and print
the SSH proof:

```bash
python scripts/wait_for_human_purchase.py \
  --context-path /tmp/market-human-buyer/context.json \
  --seller-order-id <seller-order-id> \
  --buyer-order-id <buyer-order-id>
```

This helper:

- finds the matching provisioning create job
- waits for it to reach a terminal state
- fetches buyer-visible credentials
- runs an SSH probe using `SSH_PRIVATE_KEY_PATH`
- writes a purchase artifact next to `context.json`

## 7. Clean Up The Human Purchase

Close both orders and reclaim the VM:

```bash
python scripts/cleanup_human_purchase.py \
  --context-path /tmp/market-human-buyer/context.json \
  --seller-order-id <seller-order-id> \
  --buyer-order-id <buyer-order-id> \
  --job-id <create-job-id>
```

This helper:

- runs `market order close` for the seller order
- runs `market order close` for the buyer order
- submits `destroy`
- submits `undefine`
- writes a cleanup artifact next to `context.json`

## Success Criteria

Treat the manual run as successful only when all of the following are true:

- the tunnel opens and `scripts/start_human_buyer_tunnel.py check` is green
- the sandbox CLI is installed locally
- `market order create` succeeds for the seller
- `market order match` succeeds for the buyer
- `scripts/wait_for_human_purchase.py` reaches a `succeeded` provisioning job
- the SSH probe succeeds
- `scripts/cleanup_human_purchase.py` closes both orders and reclaims the VM

## Notes

- If you want a dedicated manual buyer identity, override `BUYER_PRIVATE_KEY`
  in the local wallet bundle before running `scripts/setup_human_buyer_sandbox.py`.
- `scripts/seed_human_seller_offer.py` uses the seller's live
  `/resources/portfolio` output instead of hardcoding a GPU model. That avoids
  creating offers that the isolated seller cannot actually provision.
- Keep the tunnel terminal open until both `scripts/wait_for_human_purchase.py`
  and `scripts/cleanup_human_purchase.py` are done.
